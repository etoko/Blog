Prefix Lists

Like an access list, a prefix-list is a filtering tool; however, unlike ACLs, prefix-lists are predominantly used by routing protocols for route filtering. An ACL does not have the ability to match the subnet mask. A prefix list can provide matching of the prefix, prefix-length or both. Like an ACL, a prefix-list uses same same concept of permit or deny clauses. The naming and structure of prefix lists is similar to named ACLs. Prefix-lists use the concept of a unique name for a single prefix-list with multiple entries. Internal processing of IP prefix-lists uses an internal tree structure that results in faster matching of routes compared to ACLs. This results in faster prefix matching. Though improvements have been made with processing both prefix-lists and ACLs in hardware.

Naming and Structure

Naming and structure of prefix lists is similar to named ACLs. Prefix lists use the concept of a unique name for a single prefix-list with multiple entries. Prefix-lists use the same concept of sequence number to allow later addition or deletion of individual commands (entries) from a prefix list.

Processing Prefix Lists

Prefix-lists use the same concept of sequence number to allow later addition or deletion of individual commands (entries) from a prefix-list. A prefix-list is used to match routes particularly for route filtering and not for packet filtering:

  • Permit: the route is matched, the route should not be filtered.
  • Deny: route is not matched and should be filtered.

Prefix-list have a default deny at the end. The command to configure a prefix-list is: ip prefix-list <name> [seq <num>] {deny | permit} <prefix/prefix-length> [ge <length>] [le <length>] where:

  • ge ( greater than or equal): for the subnet mask to match, the subnet mask length must be the same as the specified prefix-length or greater i.e., between the subnet mask length and 32.

    It is important to note that the prefix-length MUST be less than the ge value; otherwise IOS will report this error message % Invalid prefix range for 10.1.1.0/24, make sure: len < ge.value <= le.value. ge value is guided by the following formular: ge <= subnet mask <= 32.

  • le (less than or equal to): for the subnet mask to match, the subnet mask length must be the same as the specified prefix-length or less i.e., between the 1 and subnet mask length.

    It is important to note that the prefix-length MUST be greater than the le value; otherwise IOS will report this error message:

    % Invalid prefix range for 10.1.1.0/24, make sure: len < ge.value <= le.value
    le value is guided by the following formula: config-length <= subnet mask <= le.

  • le and ge: if both are configured, the value of le must be greater than or equal to the value of ge:
    • If le is equal to ge: one particular subnet mask is matched.

Prefix-lists do not use wildcard masks or bits; they use the prefix length.

The logic of IP prefix-list is as follows:

  • The route's prefix must be within the range of addresses implied by the prefix-list command's prefix/length parameters.
  • The route's prefix-length must match the range of prefixes implied by the prefix-list command's prefix-length, ge and le parameters.

Prefix Matching

Every matching done by a prefix list checks for matching of both the network address and the subnet mask. If the prefix-length range is not specified (using ge and/or le), then the subnet mask and prefix length of the prefix list must match. Up to four types of matching can be performed using prefix lists:

  1. Exact match:
  2. From minimum prefix length:
  3. Up to a maximum prefix length: implemented using the le keyword permits prefixes with a subnet mask or prefix length starting from the defined length up to le value.
  4. Range of prefix lengths:

  1. ip prefix-list PL 10.1.1.0/24 A router matches a received update 10.1.1.0/24 with the prefix-list as follows:
    • Network Address: the first 24-bits of the network address are checked to see if they match with the prefix-list's prefix value of 10.1.1.0.
    • Subnet mask: the subnet mask has to be an exact match of 24 bits. the first 24 bits of the subnet mask are checked to see if they match with the prefix-lists prefix-length value. If this case it matches. An update such as 10.1.1.0 with a subnet mask of 255.255.255.128 will not match.
  2. ip prefix-list PL deny 10.1.1.0/16 ge 22 le 30
    • Matches the network address 10.1.x.x
    • Matches the subnet 255.255.252.0 - 255.255.255.252
  3. ip prefix-list PL deny 10.1.1.0/24 ge 26
    • The prefix-length of 24-bits matches against the prefix.
    • ge 26 matches against the subnet mask where the subnet mask length must be between 26 - 32.
    4. ip prefix-list PL deny 10.1.1.0/24 ge 26
  4. ip prefix-list PL deny 10.1.1.0/24 le 27
    • Matches for a network address of 10.1.1.0/24
    • The subnet mask should be 29 bits long or less i.e. 24 - 29.
  5. ip prefix-list PL deny 10.1.1.0/24 ge 26 le 29
    • Matches the network address of 10.1.1.x
    • Matches the subnet mask 255.255.255.192 - 255.255.255.248.
  6. ip prefix-list PL deny 10.1.1.0/24 ge 26 le 26
    • Matches the network address 10.1.1.x
    • Matches the subnet 255.255.255.192
  7. ip prefix-list PL deny 10.1.1.0/8 ge 9
    • Matches the network address of 10.x.x.x
    • Matches the subnet mask 255.0.0.0 - 255.128.0.0.
  8. ip prefix-list PL deny 10.1.1.0/16 le 26
    • Matches the network address 10.1.x.x
    • Matches the subnet 255.255.0.0 - 255.255.255.192

Matching the Default Route

IPv4

ip prefix-list PL permit 0.0.0.0/0

  • Match any address
  • The subnet mask MUST match zero(0). Only the default route has a subnet mask of zero.

IPv6

ipv6 prefix-list PL permit ::/0

Match all prefixes

IPv4

ip prefix-list PL permit 0.0.0.0/0 le 32

  • le 32 implies that the subnet mask will match from 0.0.0.0 - 255.255.255.255. Even the broadcast address is matched.

IPv6

ipv6 prefix-list PL permit ::/0 le 128
